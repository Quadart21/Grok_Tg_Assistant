## 1. Назначение и цели

### 1.1. Назначение

Локальное приложение с **веб-интерфейсом в браузере** для:

- массовой **холодной рассылки** в Telegram (первые сообщения по username);
- **автоматического ведения диалогов** с помощью LLM (Grok, OpenAI, Gemini, Claude, DeepSeek, OpenRouter);
- управления **пулом Telegram-аккаунтов** (tdata → `.session`, прокси, профили);
- работы **AI-ассистентов** (секретарь на отдельном аккаунте), не участвующих в общей рассылке.

### 1.2. Цели

| Цель | Критерий успеха |
|------|-----------------|
| Автоматизация outreach | Запуск рассылки на N аккаунтов × M контактов одной кнопкой |
| Память диалогов | Переписка сохраняется между перезапусками |
| Масштаб аккаунтов | Десятки сессий, фильтры, массовые операции |
| Разделение ролей | Ассистент ≠ рассылка; видны только «активные» аккаунты |
| Локальность | Работа только на `127.0.0.1`, без облачного деплоя |

### 1.3. Ограничения

- Только **личные сообщения** (username), не группы и каналы.
- Основная среда — **Windows** (Python кроссплатформенный).
- Нет многопользовательского доступа и авторизации в панели.
- Статус «активен» = наличие `.session` (без live-проверки через Telegram API при каждой загрузке списка).

```
┌─────────────────────────────────────────────────────────┐
│  Браузер (web/panel/)                                   │
│  index.html + app.js + style.css                        │
└────────────────────┬────────────────────────────────────┘
                     │ REST /api/*
┌────────────────────▼────────────────────────────────────┐
│  core/web_server.py (FastAPI)                           │

└──┬──────────────┬──────────────┬───────────────────────┘
   │              │              │
   ▼              ▼              ▼
DialogEngine   AgentEngine   TdataConverter
(outreach)     (ассистенты)  (сессии)
   │              │              │
   ▼              ▼              ▼
TelegramAccountClient + LLMClient + StateStore
   │                              │
   ▼                              ▼
sessions/                    data/state.json

```

### 4.1. Потоки выполнения

- **Outreach** (`DialogEngine`) — отдельный `threading.Thread` + `asyncio`.
- **AI-агенты** (`AgentEngine`) — отдельный поток; может работать параллельно с outreach на **других** аккаунтах.
- Оба движка пишут в общий `StateStore` и журнал (`AppService._logs`).

### 4.2. Ключевые модули

| Модуль | Файл | Назначение |
|--------|------|------------|
| Точка входа | `main.py` | Запуск веб-панели |

| Сервис | `core/app_service.py` | Оркестрация, конфиг, аккаунты |
| Outreach | `core/dialog_engine.py` | Рассылка + автоответы |

| `config/master_prompt.json` | Мастер-промпт |

| `data/state.json` | Диалоги, история сообщений, привязки |
| `web/panel/` | Веб-интерфейс панели |
| `web/guidance/` | Справочная страница |
| `docs/` | Документация (в т.ч. это ТЗ) |
| `venv/` | Виртуальное окружение Python |

### 5.2. Формат аккаунта в `sessions/`

```
sessions/
  TG_2898/                    # или номер телефона: 17755106079
    tdata/                    # исходные данные Telegram Desktop
    17755106079.session       # после конвертации
    twoFA.txt                 # опционально; приоритет над общим 2FA
```

### 5.3. Модель аккаунта (API `GET /api/accounts`)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Идентификатор (имя папки / stem `.session`) |
| `format` | `tdata` \| `session` | Тип сессии |
| `session_ready` | bool | Существует рабочий `.session` |
| `is_active` | bool | Равно `session_ready` |
| `is_assistant` | bool | Включён в `agents.json` |
| `assistant_name` | string | Имя агента |
| `outreach_eligible` | bool | `is_active && !is_assistant` |
| `is_duplicate` | bool | Суффикс `_1`, `_2` при наличии базовой папки |
| `proxy` | object? | Текущий прокси |
| `role` | string | Название роли |
| `twofa_file` | string | Имя файла 2FA, если есть |

### 5.4. Модель диалога (`data/state.json`)

- **Ключ:** `{account_id}:{username}`
- **Поля:** `status` (active / paused / closed), `auto_reply`, `goal`, `role_prompt`, `messages[]`, `replies_count`, `dialog_mode` (outreach / agent), timestamps.

### 5.5. Слои промптов для LLM

```
мастер-промпт → роль аккаунта → extra_context (рассылка / диалог / агент)
```

## 6. Функциональные требования

### 6.1. Вкладка «Рассылка»

| ID | Требование |
|----|------------|
| FR-OUT-01 | Ввод списка username (по одному на строку, с `@` или без) |
| FR-OUT-02 | Опциональный `extra_context` — подсказка для LLM при первом сообщении |
| FR-OUT-03 | Чекбокс «Вести диалог» — автоответы на входящие |
| FR-OUT-04 | Чекбокс «Продолжить прошлые диалоги» — возобновление active / paused |
| FR-OUT-05 | Кнопка **Запустить** — новые контакты + (опционально) возобновление |
| FR-OUT-06 | Кнопка **Продолжить** — только `resume_only` (без новых username) |
| FR-OUT-07 | Кнопка **Стоп** — остановка движка |
| FR-OUT-08 | Выбор аккаунтов: если не выбраны — все `outreach_eligible`; иначе только отмеченные (ассистенты исключаются) |
| FR-OUT-09 | Журнал событий в реальном времени (polling `/api/logs`) |
| FR-OUT-10 | Статистика: total / success / failed / skipped / replies / active_dialogs |
| FR-OUT-11 | Пауза между первыми сообщениями — `delay_between_messages_sec` |
| FR-OUT-12 | Параллельность — `max_concurrent_accounts` |

### 6.2. Вкладка «Подключение»

| ID | Требование |
|----|------------|
| FR-CFG-01 | Telegram API ID / Hash с [my.telegram.org/apps](https://my.telegram.org/apps) |

| FR-CFG-03 | Выбор LLM-провайдера и модели |

## 7. Бизнес-правила

| ID | Правило |
|----|---------|
| BR-01 | `outreach_eligible = session_ready ∧ ¬is_assistant` |
| BR-02 | Ассистенты не попадают в `DialogEngine` даже при ручном выборе |
| BR-03 | Дубли `TG_2898_1` скрываются в ролях, если есть `TG_2898` с `.session` |
| BR-04 | 2FA: `twoFA.txt` в папке аккаунта > общий пароль в настройках |
| BR-05 | Прокси: binding в `state.json` > `proxies.json` |
| BR-06 | При FloodWait — ожидание и retry (Telethon) |
| BR-07 | Стоп-слова в сообщении пользователя → автоответ не отправляется |
| BR-08 | Outreach и Agents могут работать одновременно на разных аккаунтах |
| BR-09 | Существующий active / paused диалог не получает повторное первое сообщение |

## 12. Сценарии использования

### 12.1. Импорт и подготовка аккаунтов

1. Распаковать RAR: `python import_session_archives.py archive.rar`
2. Указать Telegram API ID / Hash во вкладке «Подключение»
3. При необходимости — общий 2FA или `twoFA.txt` в папке аккаунта
4. Конвертировать tdata → `.session`
5. Назначить прокси (поштучно или массово)
6. Опционально — сменить профили (имена / ники)

### 12.2. Запуск кампании outreach

1. Настроить роли во вкладке «Стиль»
2. Выбрать аккаунты (или оставить все eligible)
3. Ввести username во вкладке «Рассылка»
4. Нажать **Запустить**
5. Мониторить журнал и переписки

### 12.3. Продолжение на следующий день

1. Открыть панель
2. Нажать **Продолжить** или **Запустить** с чекбоксом «Продолжить прошлые диалоги»

### 12.4. AI-секретарь

1. Вкладка «AI Агент» → добавить агента на свободный активный аккаунт
2. Аккаунт автоматически исключается из рассылки
3. **Запустить выбранных** агентов
4. Агент отвечает в личке по промпту

## 13. Ограничения и out of scope

- Нет live-проверки валидности сессии через Telegram API при загрузке списка аккаунтов.
- Нет встроенного решателя капчи и SMS-логина (только готовые tdata / `.session`).
- Нет мульти-инстанса (один процесс на машину).
- Нет экспорта отчётов в CSV / Excel.
- Нет webhook-уведомлений.
- Legacy `gui/app.py` не синхронизирован с веб-панелью.
- Нет работы с группами и каналами.
